# Phase 999.1 Plan 06: SLA de Follow-up — UI de Configuração (D-02) Summary

Componente `SlaFollowupConfig` que permite a gerente+ definir, na tela de Configurações, o prazo de SLA (em horas) por fase ativa do funil e um default global — gravando exatamente nas chaves `configuracoes.metas.sla_followup_horas_por_stage` e `sla_followup_horas_default` que o trigger SQL `upsert_lead_sla` (Plan 01) lê para calcular `deadline_at`.

## What Was Built

**Task 1 — `src/components/configuracoes/SlaFollowupConfig.tsx`:**
- Espelha o padrão canônico de `ConfigDiasAprovacao.tsx`: lê `useConfiguracoes()`, estado local sincronizado via `useEffect`, salva com `useUpdateConfiguracoes().mutateAsync`.
- `STAGES_CONFIGURAVEIS` = `PIPELINE_STAGES` filtrado por `!SLA_EXCLUDED_STAGES.includes(stage.id)` (import de `@/lib/sla-followup`, sem redeclarar a lista) — exclui `ganho_assessoria`, `ganho_consultoria`, `perdido`, `cancelado`, `stand_by`.
- Estado local: `horasPorStage: Record<string, number>` (inicializado de `metas.sla_followup_horas_por_stage`, fallback `{}`) + `horasDefault: number` (inicializado de `metas.sla_followup_horas_default`, fallback `24`).
- Renderiza uma linha por fase ativa (label de `PIPELINE_STAGES`) com um `Input` numérico de horas (placeholder mostra o default vigente quando a fase não tem override), mais uma linha separada "Default global".
- `handleSave` valida que todos os valores (default + overrides) são finitos e `> 0`; se falhar, `toast.error` e não chama a mutation.
- Salva via `updateConfigs.mutateAsync({ metas: { ...configs.metas, sla_followup_horas_por_stage: horasPorStage, sla_followup_horas_default: horasDefault } })` — merge explícito com os metas atuais, preservando `meta_leads_mes`, `dias_para_aprovacao_pendente`, pontos de gamificação, etc.
- Todo o formulário fica dentro de `<RequireRole atLeast="gerente" fallback={null}>` — consultor/coordenador não veem nem editam (a RLS `gerente_update_configuracoes`, migration 040, é a fronteira real de segurança).
- Textos em PT-BR; unidade "horas" explícita em cada linha; nota de que fases terminais e Stand By não têm SLA.

**Task 2 — Montagem em `ConfiguracoesPage.tsx`:**
- Import de `SlaFollowupConfig` ao lado do import existente de `ConfigDiasAprovacao`.
- Novo `<Card>` "SLA de follow-up" logo após o card de "Aprovação de Documentos" (mesma seção de configurações operacionais/metas), com subtítulo PT-BR explicando o propósito, seguido de `<SlaFollowupConfig />`.
- Nenhuma alteração na lógica de save pré-existente da página — o novo card salva de forma independente via seu próprio `useUpdateConfiguracoes`.
