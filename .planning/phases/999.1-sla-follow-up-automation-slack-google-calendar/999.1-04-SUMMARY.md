# Phase 999.1 Plan 04: cron-sla-scan Summary

**Fail-closed pg_cron scan (`cron_sla_scan()`) that reads the SLA webhook secret from Vault, finds warning and escalation candidates in `lead_sla` via the partial scan index, and dispatches each to `notify-sla-evento` via `pg_net` every 15 minutes — closing the loop cron → edge function → Slack/Calendar.**

## Accomplishments
- Created `supabase/migrations/043_cron_sla_scan.sql`: `public.cron_sla_scan()` SECURITY DEFINER function that mirrors the exact pattern of `038_cron_aprovacoes_stale.sql` (Vault secret lookup, fail-closed RAISE WARNING + RETURN, loop + `net.http_post`)
- Two independent scan loops in one function: warning candidates (`resolved_at IS NULL AND warning_sent_at IS NULL AND deadline_at - now() <= interval '2 hours' AND deadline_at > now()`) and escalation candidates (`resolved_at IS NULL AND escalated_at IS NULL AND deadline_at <= now()`)
- Both loops POST to `https://wfnriqwkzdazdbuzbyug.supabase.co/functions/v1/notify-sla-evento` with `{evento, lead_id}` body and `Authorization: Bearer <secret>` header, matching the `notify-sla-evento` payload contract exactly (`evento` in `['warning','escalonamento']`, `lead_id` UUID)
- `cron.schedule('sla-scan', '*/15 * * * *', 'SELECT public.cron_sla_scan()')` — idempotent by jobname, matching the ~15min frequency recommended in RESEARCH.md
- Verification grep run: 9 matches (threshold >= 5) confirming all structural elements present
