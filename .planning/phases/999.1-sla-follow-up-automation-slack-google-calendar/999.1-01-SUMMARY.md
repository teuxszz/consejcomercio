# Phase 999.1 Plan 01: Relógio do SLA (lead_sla) Summary

Tabela dedicada `lead_sla` + 2 triggers SECURITY DEFINER que recomputam o prazo por fase em mudança de status e resolvem a linha ao registrar interação, com RLS somente-leitura escopada por responsável.

## What Was Built

**Task 1 — Helper puro `src/lib/sla-followup.ts` (TDD RED→GREEN):**
- `SLA_EXCLUDED_STAGES` (as const): `ganho_assessoria`, `ganho_consultoria`, `perdido`, `cancelado`, `stand_by` — as 5 fases sem SLA.
- `SlaFollowupConfig` type: `{ porStage: Record<string, number>; default: number }`.
- `calcularDeadlineSla(stage, enteredAt, config)`: retorna `null` para fases excluídas; senão `enteredAt + horas*3600*1000`, com `horas = porStage[stage] ?? default ?? 24`.
- `slaConfigFromMetas(metas)`: extrai a config a partir de `MetasConfig`, com defaults seguros.
- `MetasConfig` (types/index.ts) ganhou `sla_followup_horas_por_stage?` e `sla_followup_horas_default?` (D-02).
- `DEFAULT_METAS` (useConfiguracoes.ts) ganhou `sla_followup_horas_default: 24` e `sla_followup_horas_por_stage: {}`.
- 11 testes cobrindo: override por fase, fallback para default global, fallback 24h quando default ausente, cada uma das 5 fases excluídas → null, extração via `slaConfigFromMetas`.

**Task 2 — Migration `supabase/migrations/041_lead_sla.sql`:**
- Tabela `lead_sla` (PK `lead_id`, `stage`, `entered_at`, `deadline_at`, `warning_sent_at`, `escalated_at`, `resolved_at`, `calendar_event_id`, timestamps). Índice parcial `idx_lead_sla_scan` (`deadline_at`) `WHERE resolved_at IS NULL`.
- `upsert_lead_sla()` (SECURITY DEFINER, `SET search_path=public`): guarda anti-reset (`OLD.status IS NOT DISTINCT FROM NEW.status` → no-op); fase terminal/stand_by → `DELETE`; senão lê `configuracoes.metas` via `COALESCE` (por-stage → default → 24) e faz `INSERT ... ON CONFLICT (lead_id) DO UPDATE` resetando os campos de estado (`warning_sent_at`/`escalated_at`/`resolved_at`/`calendar_event_id` → `NULL`).
- Trigger `trg_upsert_lead_sla`: `AFTER INSERT OR UPDATE OF status ON leads`.
- `resolve_lead_sla()` (SECURITY DEFINER): `UPDATE lead_sla SET resolved_at=now() WHERE lead_id=NEW.lead_id AND resolved_at IS NULL`.
- Trigger `trg_resolve_lead_sla`: `AFTER INSERT ON interacoes_lead`.
- RLS: `interno_role_aware_lead_sla` — `FOR SELECT` reusando `public.can_see_responsavel(leads.responsavel_id)` (migration 029). Nenhuma policy de INSERT/UPDATE/DELETE para `authenticated`.
- CHECK aditivo em `notificacoes_envios.tipo`: acrescenta `sla_warning`, `sla_escalonamento` preservando os 5 valores já existentes (`tarefa`,`cadencia`,`renovacao`,`indicacao`,`documentos`).
- **NÃO aplicada ao banco de produção** — apenas escrita/commitada, conforme instrução. Aplicação é passo manual do usuário (via Studio SQL Editor, dado o blocker de `supabase db push` já documentado em STATE.md Tech Debt).

**Task 3 — Scaffold `tests/rls/lead_sla.test.ts`:**
- Espelha o formato de `tests/rls/notificacoes_envios.test.ts` (parser de `.env.test`, `describe.skipIf(!configurado)`, tag `[RLS-TEST-041]`).
- Cobre SLA-02 (fase ativa tem linha com `deadline_at > entered_at`; `stand_by` e `ganho_assessoria` apagam a linha), SLA-03 (INSERT em `interacoes_lead` marca `resolved_at`), e a negativa de RLS (consultor não INSERT/UPDATE direto; SELECT escopado).
- Executa PULADO (sem `.env.test` configurado no ambiente local) — 1 passed / 7 skipped, sem erro de parse.
