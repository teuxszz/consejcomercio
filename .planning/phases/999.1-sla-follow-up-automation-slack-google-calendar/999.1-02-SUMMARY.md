# Phase 999.1 Plan 02: Cofre de Credenciais do Google Calendar Summary

Tabela `google_calendar_tokens` com RLS zero-leitura (nem o dono lê), 2 RPCs SECURITY DEFINER que expõem só status/disconnect, e o helper de Edge Function `_shared/google-calendar.ts` que refresca o access_token e cria o bloco de follow-up via REST cru (sem SDK).

## What Was Built

**Task 1 — Migration `supabase/migrations/042_google_calendar_tokens.sql`:**
- Tabela `google_calendar_tokens` (PK `perfil_id` → `perfis(id) ON DELETE CASCADE`, `refresh_token`, `access_token`, `access_token_expires_at`, `scope`, timestamps).
- `ENABLE ROW LEVEL SECURITY` com **nenhuma policy `FOR SELECT`** para `authenticated` — o Postgres nega toda leitura por padrão sem essa policy (T-999.1-01). Único leitor legítimo do token é `service_role` (bypassa RLS).
- Policies `own_row_insert` (`FOR INSERT ... WITH CHECK (perfil_id = auth.uid())`) e `own_row_update` (`FOR UPDATE ... USING/WITH CHECK (perfil_id = auth.uid())`) — escrita escopada (T-999.1-08). Sem policy `DELETE`.
- RPC `google_calendar_status()` (`LANGUAGE sql STABLE SECURITY DEFINER SET search_path=public, pg_temp`) — devolve `TABLE(conectado boolean, expira_em timestamptz)` via `EXISTS(...)` + `access_token_expires_at`, nunca o token. `GRANT EXECUTE TO authenticated`.
- RPC `google_calendar_disconnect()` (mesmo padrão `SECURITY DEFINER`) — `DELETE FROM google_calendar_tokens WHERE perfil_id = auth.uid()`. `GRANT EXECUTE TO authenticated`.
- **NÃO aplicada ao banco de produção** — apenas escrita/commitada, conforme instrução explícita da tarefa.

**Task 2 — `supabase/functions/_shared/google-calendar.ts` (TDD RED→GREEN):**
- `refreshAccessToken(refreshToken)`: `POST` `x-www-form-urlencoded` para `https://oauth2.googleapis.com/token` (`grant_type=refresh_token` + `client_id`/`client_secret` de `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET`). Non-2xx (ex.: `invalid_grant`) → `null`; nunca loga token/secret (T-999.1-04, T-999.1-06).
- `createFollowupEvent(accessToken, leadNome, deadlineAt, leadId, appUrl)`: `POST` `calendars/primary/events` — bloco de 30min terminando no `deadlineAt`, `timeZone: America/Sao_Paulo`, `summary: "Fazer follow-up com Lead <nome>"`, `description` com deep link `<appUrl>/leads/<leadId>` (Open Q4 — incluído por padrão) + nota "Gerado pelo CONSEJ CRM — SLA de follow-up". Non-2xx → `{ ok: false, status }`.
- `getValidAccessToken(supabase, perfilId)`: lê a linha via cliente injetado (service_role no caller real); sem linha → `null`; `access_token` válido com margem de ~2min → devolve sem chamar refresh; expirado → `refreshAccessToken`, persiste (`UPDATE access_token/access_token_expires_at/updated_at`), devolve; refresh `null` → devolve `null` (fallback Slack-only, Plan 03).
- Teste `google-calendar.test.ts`: 8 casos (2 `refreshAccessToken`, 2 `createFollowupEvent`, 4 `getValidAccessToken` incluindo "sem linha" e "refresh falha"), mockando `globalThis.fetch` via `vi.spyOn` — nenhuma chamada real ao Google. RED confirmado (import falhava — módulo não existia) antes da implementação.

**Task 3 — Scaffold `tests/rls/google_calendar_tokens.test.ts`:**
- Espelha `tests/rls/notificacoes_envios.test.ts` / `tests/rls/lead_sla.test.ts` (parser `.env.test`, `describe.skipIf(!configurado)`, tag `[RLS-TEST-042]`).
- Semeia via `service_role` uma linha de token dummy para o consultor.
- Cobre: (a) SLA-06 — consultor **e** diretor autenticados NÃO conseguem `SELECT`, nem da própria linha; (b) consultor consegue `UPDATE` a própria linha mas NÃO consegue `INSERT`/`UPDATE` linha de outro usuário (bloqueado por `WITH CHECK`/`USING`); (c) RPC `google_calendar_status()` devolve shape `{ conectado, expira_em }` — `true`/expiry quando conectado, `false`/`null` quando nunca conectou — nunca expõe token.
- Executa PULADO neste ambiente (sem `.env.test`) — 1 passed / 7 skipped, sem erro de parse.
