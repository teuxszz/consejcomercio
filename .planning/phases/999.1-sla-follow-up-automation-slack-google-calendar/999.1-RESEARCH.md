# Phase 999.1: Automação de SLA de Follow-up (Slack DM + Google Agenda + Escalação) - Research

**Researched:** 2026-07-14
**Domain:** Postgres pg_cron/pg_net automation engine + Slack DM (existing) + Google Calendar API OAuth 2.0 per-user (new integration)
**Confidence:** MEDIUM-HIGH — the notification/cron/RLS mechanics are HIGH confidence (100% derived from working code already in this repo); the Google Calendar OAuth flow is MEDIUM confidence (verified against current Google/Supabase docs this session, but not yet exercised against this specific Supabase project)

## Summary

This phase has **no genuinely new "notification plumbing" to invent** — the repo already has a mature, battle-tested pattern for "cron scans DB → pg_net POSTs to an edge function → edge function DMs Slack (+ email/push) → idempotent log" used by `notify-renovacao` (031), `notify-resumo-diario` (034), and `notify-aprovacao-evento` / `notify-aprovacoes-stale` (038). The SLA engine should be built as a **new instance of this exact pattern**, not a new architecture. The only genuinely new piece is **Google Calendar OAuth per-user token storage and refresh**, which does not exist anywhere in this codebase today.

The recommended design has three independently testable parts:

1. **SLA state table** (`lead_sla`) populated by a trigger on `leads` (fires on INSERT and on `UPDATE OF status`) that computes a `deadline_at` from a configurable per-stage hour budget stored in `configuracoes.metas` (same JSONB-config pattern used for `dias_para_aprovacao_pendente` in migration 038). A second trigger on `interacoes_lead` marks the row `resolved_at` the moment the assessor logs contact — this is the "assessor registrou atividade" signal the phase description asks for.
2. **pg_cron scan every 15 min** (mirrors 034/038's `cron.schedule` + `pg_net.http_post` pattern) that finds rows within 2h of `deadline_at` (fires the warning: Calendar block + Slack DM) and rows past `deadline_at` unresolved (fires escalation to the gerência Slack channel). Idempotency uses a **compare-and-swap `UPDATE ... WHERE x_sent_at IS NULL RETURNING`** inside the edge function — stronger than the existing `INSERT ... ON CONFLICT` idempotency pattern because this cron runs every 15 min (not once/day), so the race window matters more here than it did for `notify-renovacao`.
3. **Google Calendar integration** via **Supabase Auth's built-in `linkIdentity()` with the Google provider** (not a hand-rolled OAuth redirect/exchange edge function). This lets the assessor authorize Calendar access with one click while staying logged in via email/password; Supabase's GoTrue server performs the code exchange using Google credentials configured once in the Supabase Dashboard, and momentarily surfaces `provider_token`/`provider_refresh_token` in the client session immediately after the redirect completes. The SPA persists these into a new locked-down table (`google_calendar_tokens`) in that same instant — this is the only window in which the refresh token is ever visible to the browser.

**Primary recommendation:** Reuse the existing SQL-trigger + `pg_cron`/`pg_net` + `_shared/slack.ts` DM pattern verbatim for the SLA engine (do not invent a new dispatch mechanism); for Google Calendar, use Supabase's native `linkIdentity()` + raw `fetch` to the Calendar REST API (no `googleapis` SDK) — this avoids a new npm dependency entirely and keeps the edge function bundle Deno-friendly like `_shared/slack.ts`.

### Pattern 1: SQL trigger computes SLA deadline on stage entry (mirrors `update_updated_at`/`cliente_docs_after_change`)

**What:** A `SECURITY DEFINER` trigger function on `leads` recomputes `lead_sla` whenever a lead is created or its `status` changes.
**When to use:** Any time the "clock" for a per-stage SLA needs to restart deterministically, without relying on the general-purpose `leads.updated_at` (which changes on *any* field edit, not just stage changes — confirmed via `001_initial_schema.sql`'s generic `update_updated_at()` trigger, so it **cannot** be reused as the SLA clock start).

```sql
-- Source: pattern derived from supabase/migrations/038_cron_aprovacoes_stale.sql
-- (cliente_docs_after_change) — same "hydrate → decide event → write" shape.
CREATE OR REPLACE FUNCTION public.upsert_lead_sla()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_horas NUMERIC;
BEGIN
  -- Terminal ou stand_by: sem SLA (mesma exclusão de 034_cron_resumo_diario.sql)
  IF NEW.status IN ('ganho_assessoria','ganho_consultoria','perdido','cancelado','stand_by') THEN
    DELETE FROM lead_sla WHERE lead_id = NEW.id;
    RETURN NEW;
  END IF;

  -- Só recalcula se é INSERT ou se o status mudou de fato
  IF TG_OP = 'UPDATE' AND OLD.status IS NOT DISTINCT FROM NEW.status THEN
    RETURN NEW;
  END IF;

  SELECT COALESCE(
    (metas->'sla_followup_horas_por_stage'->>NEW.status)::numeric,
    (metas->>'sla_followup_horas_default')::numeric,
    24
  ) INTO v_horas
  FROM configuracoes WHERE id = 'default';

  INSERT INTO lead_sla (lead_id, stage, entered_at, deadline_at)
  VALUES (NEW.id, NEW.status, now(), now() + (v_horas || ' hours')::interval)
  ON CONFLICT (lead_id) DO UPDATE SET
    stage = EXCLUDED.stage,
    entered_at = EXCLUDED.entered_at,
    deadline_at = EXCLUDED.deadline_at,
    warning_sent_at = NULL,
    escalated_at = NULL,
    resolved_at = NULL,
    calendar_event_id = NULL;

  RETURN NEW;
END;
$$;
```

### Pattern 2: Idempotent compare-and-swap dispatch (stronger than existing 031/038 pattern)

**What:** Instead of the INSERT-then-check-23505 idempotency used by `_shared/email.ts`/`_shared/push.ts` (fine for once-a-day crons), use an atomic `UPDATE ... WHERE column IS NULL RETURNING` so a 15-minute cron cannot double-send even under retry/overlap.
**When to use:** Any cron/webhook running more frequently than daily where the dispatch side-effect (Slack DM, Calendar event) is not itself idempotent.

```typescript
// Source: pattern proposed for supabase/functions/_shared/sla.ts
// (no direct precedent in repo — existing crons run 1x/day so the race window
// there is negligible; this cron runs every 15min so CAS is warranted)
const { data: claimed, error } = await supabase
  .from('lead_sla')
  .update({ warning_sent_at: new Date().toISOString() })
  .eq('lead_id', payload.lead_id)
  .is('warning_sent_at', null)
  .select('lead_id')
  .maybeSingle()

if (!claimed) {
  // Já processado por outra invocação concorrente (ou cron duplicado)
  return json({ ok: true, skipped: 'already_sent' })
}
// ... segue com Slack DM + Calendar (só quem "ganhou" a CAS chega aqui)
```

### Pattern 3: Google Calendar OAuth via Supabase `linkIdentity` (no custom callback route)

**What:** The assessor clicks "Conectar Google Agenda" while already logged in via email/password. Supabase's GoTrue server (not our code) performs the redirect + code exchange using Google OAuth credentials configured once in the Supabase Dashboard. The browser briefly sees `provider_token`/`provider_refresh_token` in the session right after the redirect completes — this is the *only* moment those values are ever client-visible, and they must be persisted immediately.

**When to use:** Any per-user Google API access where the app already uses Supabase Auth as primary login (this project's exact situation) and does not need Google as a login method itself.

```typescript
// Source: https://supabase.com/docs/guides/auth/social-login/auth-google
// + https://supabase.com/docs/guides/auth/auth-identity-linking
// (verified this session — see Sources)

// 1. Trigger the OAuth dance (button click in ConectarGoogleAgendaCard.tsx)
const { error } = await supabase.auth.linkIdentity({
  provider: 'google',
  options: {
    redirectTo: `${window.location.origin}/me/preferencias?google_linked=1`,
    scopes: 'https://www.googleapis.com/auth/calendar.events',
    queryParams: {
      access_type: 'offline', // obrigatório para receber provider_refresh_token
      prompt: 'consent',      // obrigatório para RE-emitir refresh_token em reconexões
    },
  },
})

// 2. On return (useEffect on the target page, gated by ?google_linked=1):
const { data: { session } } = await supabase.auth.getSession()
if (session?.provider_token && session?.provider_refresh_token) {
  await supabase.from('google_calendar_tokens').upsert({
    perfil_id: session.user.id,
    refresh_token: session.provider_refresh_token,
    access_token: session.provider_token,
    access_token_expires_at: new Date(Date.now() + 3500 * 1000).toISOString(), // ~58min
    scope: 'https://www.googleapis.com/auth/calendar.events',
  })
  // Limpar o query param da URL para não reprocessar em refresh (Pitfall 3)
}
```

**Required one-time manual setup (not code — document as pre-req, mirrors Phase 6's VAPID key setup):**
1. Google Cloud Console: create OAuth 2.0 Client ID (type "Web application"); Authorized redirect URI = `https://<project-ref>.supabase.co/auth/v1/callback` (Supabase's callback, **not** the CRM's own URL) `[CITED: developers.google.com/identity/protocols/oauth2/web-server]`.
2. OAuth consent screen: External user type (assessors use personal/Workspace Gmail, not a Workspace the user administers per the locked design constraint); can remain in "Testing" only if refresh-token 7-day expiry is acceptable — **it is not** (see Pitfall 1) — must move to **"In production"** publishing status. `calendar.events` is a *sensitive* (not *restricted*) scope, so under 100 users the app can run unverified in production with an "unverified app" warning screen the assessor must click through once `[CITED: developers.google.com/identity/protocols/oauth2/production-readiness/sensitive-scope-verification]`.
3. Supabase Dashboard → Authentication → Providers → Google: paste the same Client ID/Secret from step 1.
4. Supabase Dashboard → Authentication → (linking) settings: **enable "Manual Linking"** — `linkIdentity()` fails without this `[CITED: supabase.com/docs/guides/auth/auth-identity-linking]`.
5. Supabase Edge Function secrets: set `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` to the **exact same values** as step 1/3 — the edge function needs them to call the token-refresh endpoint directly (Supabase does not auto-refresh third-party provider tokens on your behalf).

### Anti-Patterns to Avoid
- **Using `leads.updated_at` as the SLA clock start:** it's touched by the generic `update_updated_at()` trigger on *any* column edit (notes, telefone, etc.), not just `status`. Use a dedicated `entered_at` in `lead_sla` instead.
- **Storing the Google refresh token anywhere client-readable:** no `SELECT` RLS policy for `authenticated` on `google_calendar_tokens` — mirrors the existing "no permissive INSERT/UPDATE policy" philosophy on `notificacoes_envios` (040 hardening explicitly closed several `USING(true)` holes; do not reopen an equivalent hole here).
- **Reading `session.provider_refresh_token` more than once:** it is only populated in the session object returned immediately after the OAuth redirect completes. A page refresh loses it permanently (must re-run `linkIdentity` with `prompt=consent` to get a new one) — capture-and-persist in the very first render after redirect.
- **Letting the cron SQL function call `net.http_post` for a lead that already has `warning_sent_at` set:** always filter `WHERE warning_sent_at IS NULL` in the cron's `SELECT`, and re-verify via CAS `UPDATE` inside the edge function — defense in depth against overlapping cron ticks.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Google OAuth redirect/state/CSRF handling | Custom `/api/google-oauth-callback` edge function | `supabase.auth.linkIdentity()` | Supabase's GoTrue already implements this correctly (state, PKCE where applicable, identity linking to the existing logged-in user) — reinventing it is pure risk for zero benefit given this project's architecture (browser talks directly to Supabase, no custom API layer). |
| Slack DM delivery + retry | New Slack HTTP client | `_shared/slack.ts` (`openDmChannel`/`postDm`) | Already handles the `conversations.open` → `chat.postMessage` two-step + exponential retry on 429/5xx. |
| Webhook auth for the new cron-triggered edge function | Custom header check | `_shared/auth.ts` (`constantTimeAuthCheck`) + Vault secret | Exact fail-closed pattern hardened in migration 040; reuse verbatim, including the "no secret configured → refuse, don't skip auth" behavior. |
| Idempotent notification bookkeeping | New ad-hoc table | Extend the existing `lead_sla` row's own `warning_sent_at`/`escalated_at` columns (self-contained state machine) rather than trying to force this into `notificacoes_envios`'s `tipo` CHECK enum, which is oriented around the email/Slack/push preference matrix, not around a stage-clock model | `notificacoes_envios` idempotency is keyed on (perfil, tipo, canal, **day**) — wrong granularity for a 15-min-resolution SLA clock. Keep the two concerns separate. |

**Key insight:** Every dispatch mechanism this phase needs (cron scan → HTTP POST → edge function → Slack DM → idempotent log) already has a working, hardened reference implementation in this repo. The only truly new muscle is Google OAuth token lifecycle management, and even there, Supabase's own Auth system removes most of the custom-code burden.

## Common Pitfalls

### Pitfall 1: Refresh tokens expire in 7 days if the Google Cloud project is left in "Testing" publishing status
**What goes wrong:** Assessor connects Google Calendar, everything works for a week, then silently stops creating events — `getValidAccessToken()` fails with `invalid_grant` and there's no obvious reason why.
**Why it happens:** Google automatically revokes refresh tokens issued by unverified/testing apps after exactly 7 days, regardless of usage `[VERIFIED: developers.google.com/identity/protocols/oauth2 — cross-confirmed via community reports]`.
**How to avoid:** Move the Google Cloud OAuth consent screen to **"In production"** publishing status (still fine to stay *unverified* since `calendar.events` is a sensitive, not restricted, scope, and CONSEJ's team is well under the 100-user cap) — production status gives indefinite refresh token lifetime.
**Warning signs:** `invalid_grant` errors appearing in edge function logs roughly 7 days after each assessor's first connection.

### Pitfall 2: `provider_refresh_token` is only visible once, immediately after the OAuth redirect
**What goes wrong:** A naive implementation reads `session.provider_refresh_token` from a generic `onAuthStateChange` listener that also fires on normal page loads/token refreshes, and finds it `undefined` most of the time, concluding (incorrectly) that the integration is broken.
**Why it happens:** Supabase only surfaces `provider_token`/`provider_refresh_token` in the session object returned by the OAuth completion event itself — subsequent `getSession()` calls do not carry them forward `[CITED: supabase.com/docs/guides/auth/social-login/auth-google]`.
**How to avoid:** Gate the capture-and-persist logic behind the `?google_linked=1` redirect query param set in the `linkIdentity` call's `redirectTo`, so it only runs exactly once, right after the round trip.
**Warning signs:** `google_calendar_tokens` row never gets created despite the user clicking through the Google consent screen successfully.

### Pitfall 3: Client-side token refresh calls fail because `client_secret` mismatches the one that issued the refresh token
**What goes wrong:** Edge function's `oauth2.googleapis.com/token` refresh call returns `invalid_client`.
**Why it happens:** The refresh token was issued during a code exchange performed by **Supabase's GoTrue server**, using the Client ID/Secret configured in the Supabase Dashboard. If the edge function's `GOOGLE_CLIENT_SECRET` Supabase secret is a *different* value (e.g., a second OAuth client created "for the API" by mistake), the refresh call is rejected — refresh tokens are bound to the exact `client_id` that requested them.
**How to avoid:** Use the identical Client ID/Secret pair in both places (Supabase Dashboard Google provider config, and Supabase Edge Function secrets) — document this explicitly as a single source of truth.
**Warning signs:** First-ever refresh attempt (roughly 1h after connecting, when the initial `access_token` expires) fails immediately.

### Pitfall 4: pg_cron running every 15 minutes multiplies the "am I first" race far more than the existing daily crons
**What goes wrong:** Two overlapping cron ticks (e.g., one is slow due to a `pg_net` backlog) both find the same `lead_sla` row eligible and both fire a Slack DM + Calendar event — duplicate noise, and possibly duplicate calendar blocks.
**Why it happens:** The existing 031/034/038 crons run once daily, so even an INSERT-then-check-23505 idempotency pattern has a negligible race window relative to the 24h period. A 15-minute period is ~96x tighter.
**How to avoid:** Use the CAS `UPDATE ... WHERE warning_sent_at IS NULL RETURNING` pattern (Pattern 2 above) as the very first statement inside the edge function, before any Slack/Calendar call — treat "0 rows returned" as "someone else already claimed this."
**Warning signs:** Duplicate Slack DMs or duplicate Calendar events for the same lead within the same 15-minute window.

### Pitfall 5: Stage re-entry mid-cycle should reset the SLA, but a no-op `UPDATE` (same status re-saved) should not
**What goes wrong:** If the trigger recomputes `lead_sla` on every `UPDATE` to `leads` regardless of whether `status` actually changed, a benign edit (e.g., fixing a typo in `notas`) silently resets the assessor's SLA clock, hiding an overdue follow-up.
**Why it happens:** `leads` has one generic `updated_at`-style trigger already; a naive SLA trigger written as `AFTER UPDATE ON leads` (without `OF status`) or without an `OLD.status IS NOT DISTINCT FROM NEW.status` guard will fire on every field edit.
**How to avoid:** Declare the trigger as `AFTER UPDATE OF status ON leads` **and** keep the `IF OLD.status IS NOT DISTINCT FROM NEW.status THEN RETURN NEW` guard for defense in depth (Postgres `UPDATE OF <col>` still fires even if the new value equals the old one when the column is named in the `SET` clause).
**Warning signs:** SLA deadlines observed to reset without a visible pipeline stage change in the Kanban.

### Pitfall 6: `stand_by` and terminal stages must be excluded, or the team will get spurious escalations for paused/closed leads
**What goes wrong:** A lead intentionally paused (`stand_by`) or already won/lost keeps accumulating `lead_sla` rows and eventually triggers a management-channel escalation for a deal that's already closed.
**Why it happens:** `stand_by` is not in `TERMINAL_STAGES` (`src/lib/constants.ts`) but the team's existing convention (`034_cron_resumo_diario.sql`'s explicit comment + `ACTIVE_LEAD_STAGES` filter in `constants.ts`) already treats it as "no cadence pressure while paused."
**How to avoid:** Mirror the exact exclusion list already used for the daily cadence cron: `('ganho_assessoria','ganho_consultoria','perdido','cancelado','stand_by')`.
**Warning signs:** Management channel receives an SLA escalation for a lead visibly marked "Stand By" or already won/lost in the Kanban.

### Cron scan SQL (mirrors 034/038 exactly)
```sql
-- Source: pattern from supabase/migrations/038_cron_aprovacoes_stale.sql
-- cron_disparar_aprovacoes_stale() — same secret-lookup + loop + net.http_post shape
CREATE OR REPLACE FUNCTION public.cron_sla_scan()
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_secret TEXT;
  v_url    TEXT := 'https://wfnriqwkzdazdbuzbyug.supabase.co/functions/v1/notify-sla-evento';
  r        RECORD;
BEGIN
  SELECT decrypted_secret INTO v_secret
    FROM vault.decrypted_secrets WHERE name = 'webhook_sla_secret';
  IF v_secret IS NULL THEN
    RAISE WARNING 'cron_sla_scan: secret webhook_sla_secret ausente no Vault';
    RETURN;
  END IF;

  -- Aviso: faltando <=2h, ainda não avisado
  FOR r IN
    SELECT ls.lead_id, l.responsavel_id, l.nome, l.empresa, ls.deadline_at
      FROM lead_sla ls JOIN leads l ON l.id = ls.lead_id
     WHERE ls.resolved_at IS NULL
       AND ls.warning_sent_at IS NULL
       AND ls.deadline_at - now() <= interval '2 hours'
       AND ls.deadline_at > now()
  LOOP
    PERFORM net.http_post(
      url := v_url,
      body := jsonb_build_object('evento','warning','lead_id',r.lead_id),
      headers := jsonb_build_object('Content-Type','application/json','Authorization','Bearer '||v_secret)
    );
  END LOOP;

  -- Escalação: vencido, ainda não escalado
  FOR r IN
    SELECT ls.lead_id, l.responsavel_id, l.nome, l.empresa, ls.deadline_at
      FROM lead_sla ls JOIN leads l ON l.id = ls.lead_id
     WHERE ls.resolved_at IS NULL
       AND ls.escalated_at IS NULL
       AND ls.deadline_at <= now()
  LOOP
    PERFORM net.http_post(
      url := v_url,
      body := jsonb_build_object('evento','escalonamento','lead_id',r.lead_id),
      headers := jsonb_build_object('Content-Type','application/json','Authorization','Bearer '||v_secret)
    );
  END LOOP;
END;
$$;

SELECT cron.schedule('sla-scan', '*/15 * * * *', 'SELECT public.cron_sla_scan()');
```

```typescript
async function createFollowupEvent(accessToken: string, leadNome: string, deadlineAt: Date) {
  const start = new Date(deadlineAt.getTime() - 30 * 60 * 1000) // bloco de 30min terminando no deadline
  const res = await fetch('https://www.googleapis.com/calendar/v3/calendars/primary/events', {

    },

    }),
  })
```

### Google OAuth token refresh
```typescript
// Source: https://developers.google.com/identity/protocols/oauth2/web-server (verified this session)
async function refreshAccessToken(refreshToken: string): Promise<{ accessToken: string; expiresAt: Date } | null> {
  const res = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      grant_type: 'refresh_token',
      refresh_token: refreshToken,
      client_id: Deno.env.get('GOOGLE_CLIENT_ID')!,
      client_secret: Deno.env.get('GOOGLE_CLIENT_SECRET')!,
    }),
  })
  if (!res.ok) return null // invalid_grant → token revogado; caller deve tratar como "desconectado"
  const body = await res.json() as { access_token: string; expires_in: number }
  return { accessToken: body.access_token, expiresAt: new Date(Date.now() + body.expires_in * 1000) }
}
```

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| XSS-driven exfiltration of Google refresh token via a compromised browser session | Information Disclosure | No `SELECT` RLS policy on `google_calendar_tokens` for `authenticated` at all — client can only write its own row, never read it back; status exposed only via a `SECURITY DEFINER` RPC returning boolean+expiry |
