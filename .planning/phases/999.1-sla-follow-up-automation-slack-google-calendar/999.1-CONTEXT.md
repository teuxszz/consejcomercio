<domain>
## Phase Boundary

Motor de SLA de follow-up que tira o gerente do papel de "fiscal manual". Quando um lead entra no funil ou muda de fase, calcula o prazo-limite (SLA) para o próximo contato. Faltando 2h, se o assessor não registrou atividade: (1) cria bloco na Google Agenda do assessor + (2) DM privada no Slack. Se vencer sem ação: escala para o canal da gerência no Slack.
</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Google Calendar — auth
- **OAuth POR USUÁRIO** via `linkIdentity()` do Supabase (GoTrue broker), com `access_type: 'offline'` + `prompt: 'consent'`. NÃO usar domain-wide delegation (o usuário não tem admin do Google Workspace).
- Refresh token de cada assessor guardado em nova tabela `google_calendar_tokens` com **RLS zero** (nem SELECT do próprio dono) — acesso só via RPC `SECURITY DEFINER`. É credencial de terceiro.
- **Pitfall obrigatório:** a tela de consentimento OAuth no Google Cloud DEVE estar em "In production" (senão refresh token expira em 7 dias). Escopo `calendar.events` é "sensitive" (não "restricted") → ok ficar unverified sob 100 usuários.
- Se o assessor ainda NÃO autorizou o Google → **fallback só-Slack** (manda a DM, pula o bloco na agenda).

### Modelo de SLA
- **POR FASE DO FUNIL** — cada estágio do pipeline tem seu próprio prazo (horas). Configurável (o usuário define os valores nas configurações; ex: em `configuracoes.metas` ou tabela dedicada — decisão do planner).
- Relógio do SLA: NÃO usar `leads.updated_at` (é tocado por qualquer edição). Usar tabela dedicada `lead_sla` com `entered_at` / `deadline_at` por lead+fase.

### Alerta (DM 2h antes)
- **SEMPRE ENVIA** — ignora `perfis.preferencias_notif`. SLA é crítico pro negócio; não pode ser silenciado por opt-in.

### Escalação (SLA vencido → gerência)
- Canal da gerência: **DEFERIDO** — o usuário ainda não tem o ID do canal (vai solicitar aos superiores). O plano deve tratar o canal como um **secret configurável** (ex: `SLACK_GERENCIA_CHANNEL_ID`). Se o secret não estiver setado, a escalação faz **fail-safe** (loga/pula sem quebrar), NÃO fail-open.

### Google Agenda — ciclo de vida do evento
- Ao RESOLVER o follow-up após o alerta: **NÃO apagar** o bloco da agenda (MVP). Guardar `calendar_event_id` para um enhancement futuro (auto-delete).

### Reuso (não reinventar)
- Motor de dispatch = nova instância do padrão existente: trigger SQL → `pg_cron`/`pg_net` → edge function → `_shared/slack.ts` (DM). Referências: `notify-renovacao` (031), `notify-resumo-diario` (034), `notify-aprovacoes-stale` (038).
- Auth de cron via secret do Vault no header (fail-closed), como as notify-* corrigidas na 040.
- Log em `notificacoes_envios`. Idempotência do scan de 15min via CAS atômico (`UPDATE ... WHERE x_sent_at IS NULL RETURNING`), NÃO só o índice único diário.
- `perfis.slack_user_id` mapeia assessor → Slack.

### Claude's Discretion
- Frequência exata do cron de scan (research sugeriu ~15min).
- Estrutura exata da config de SLA por fase (JSONB em configuracoes vs tabela).
- UI: botão "Conectar Google Agenda" em /me + tela de config dos prazos por fase.
</decisions>
