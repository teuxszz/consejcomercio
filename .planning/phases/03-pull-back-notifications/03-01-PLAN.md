## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase DB → Edge Function | Webhook envia payload via HTTPS; auth via Bearer no header `Authorization` |
| Edge Function → Slack API | Edge function autentica no Slack com `SLACK_BOT_TOKEN` (env var) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Spoofing | POST para `/functions/v1/notify-tarefa` por terceiros | mitigate | `constantTimeAuthCheck` (já implementado em notify-tarefa linhas 40-49) compara Bearer contra `WEBHOOK_TAREFA_SECRET` em tempo constante. Smoke test confirma que webhook real usa o secret correto; chamadas sem Bearer devem retornar 401. |
| T-03-01-02 | Information Disclosure | Vazamento do secret WEBHOOK_TAREFA_SECRET em logs ou no arquivo de registro | mitigate | O arquivo `03-01-NOTIF01-WEBHOOK-CHECK.md` registra APENAS o NOME do secret (`WEBHOOK_TAREFA_SECRET`), nunca o valor. O dashboard Supabase já mascara o valor após configuração. |
| T-03-01-03 | Tampering | Modificação do body do webhook em trânsito | accept | HTTPS termina em Supabase → `*.supabase.co` (TLS válido). Sem necessidade de assinatura adicional para Milestone 2; risco aceitável dado escopo interno do CRM (2-5 usuários). |
| T-03-01-04 | Spoofing | DM enviada para o usuário errado | mitigate | `notify-tarefa` resolve `slack_user_id` via `findSlackUserId(record.atribuido_a_id)` lendo `perfis` com service role. `atribuido_a_id` vem do INSERT/UPDATE real da tabela, não de input externo do webhook. |
</threat_model>
