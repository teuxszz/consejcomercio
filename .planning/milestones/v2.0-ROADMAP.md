## Overview

Milestone 2 transformou o CRM de "lugar onde deveria reportar" em "lugar onde o trabalho acontece". Quatro fases entregaram em sequência:

1. **Fundação segura** — RLS fechada + login tracking + sistema de tarefas + sinal de adoção
2. **Guia de cadência + WhatsApp** — Kanban com D-point + Vista "Ação Hoje" + botões wa.me
3. **Pull-back notifications** — Slack DM em atribuição/resumo/cadência via pg_cron
4. **Dashboard de receita** — MRR, ARR, forecast 3m, renovações 30/60/90, drill-down /contratos

Cada fase é um slice vertical usável — o time ganhou valor real ao final de cada uma.

## Milestone Summary

### Key Decisions

| Decisão | Rationale |
|---------|-----------|
| WhatsApp via wa.me (sem API) | Meta Business API exigiria número dedicado + custo fixo — incompatível com "zero custo incremental" |
| Notificações via Slack edge functions | Reúso da infraestrutura existente (Phase 1 do Milestone 1) — zero novo custo |
| DM consolidada (não por evento) com cron + edge function | Um único code path para NOTIF-02 + NOTIF-03; evita N DMs por dia |
| `stand_by` incluído no NOT IN do cron 034 | Leads pausados não devem ser cobrados na cadência; alinhado com `ACTIVE_LEAD_STAGES` |
| Revenue dashboard depende de Phase 1 (adoção) | Dados incompletos tornam analytics inútil — primeiro consertar friction, depois métricas |
| MRR pro-rata em `valor_total ÷ duração` (D-01) | Reflete receita "como se" o contrato fosse mensal — comparável com assessoria recorrente |
| `valor_protegido` NÃO contribui ao MRR (D-01) | Provisão de risco, não receita realizada |
| Forecast contratos sem `data_fim` não entram em saídas (D-03) | Pitfall — sem data_fim significa "renova automaticamente"; tratá-los como saída inflaria churn |
